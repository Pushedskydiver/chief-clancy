# Glossary

Ubiquitous language for the Clancy project. Use these terms consistently in code, docs, commits, and agent prompts.

## Monorepo

| Term                          | Definition                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| ----------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Core**                      | The `@chief-clancy/core` package. Domain model — `board/`, `types/`, `schemas/`, `shared/`. Previously incubated capabilities (`dev`, `brief`, `plan`) have since been extracted into standalone workspace packages.                                                                                                                                                                                                                                                                                                                           |
| **Terminal**                  | The `@chief-clancy/terminal` package. CLI-facing surface — `installer/`, `runner/`, `roles/`, `agents/`, `templates/`, `hooks/`, `entrypoints/`, `shared/`. Depends on core and dev.                                                                                                                                                                                                                                                                                                                                                           |
| **Wrapper**                   | The `chief-clancy` package. Thin bin entry point that delegates to terminal. What users install via `npx chief-clancy`.                                                                                                                                                                                                                                                                                                                                                                                                                        |
| **Capability directory**      | Historically, a directory within core/terminal grouping one capability as a candidate for package extraction (`dev/`, `brief/`, `plan/`, etc.). The pattern produced the current set of standalone packages; what remains as capability-shaped directories today lives inside individual packages (e.g. terminal's `installer/`, `runner/`, `roles/`).                                                                                                                                                                                         |
| **Extraction criteria**       | When to promote a capability directory to its own package: a second consumer exists, the directory exceeds 2000 lines, or it needs independent release cadence.                                                                                                                                                                                                                                                                                                                                                                                |
| **Workspace**                 | A pnpm workspace — each directory under `packages/` is a workspace with its own `package.json`.                                                                                                                                                                                                                                                                                                                                                                                                                                                |
| **Barrel export**             | An `index.ts` file that re-exports symbols from sibling files. Post-[PR #305](https://github.com/Pushedskydiver/chief-clancy/pull/305) (dev/terminal) and Barrier-Core (which first shipped core at `1.0.0`; core has since bumped to `4.0.0`), internal barrels exist only at package entry points (`src/index.ts`). See [CONVENTIONS.md §Folder Structure](CONVENTIONS.md#folder-structure) for the five-category model.                                                                                                                     |
| **Wrapper folder**            | A folder grouping ≥2 source files (tests don't count) that implement a single concept (e.g. `lifecycle/rework/` — `rework.ts` + `rework-builders.ts` + `rework-handlers.ts`). Single-file concepts stay flat.                                                                                                                                                                                                                                                                                                                                  |
| **Grouping folder**           | A folder clustering multiple related concepts under a ubiquitous-language name (e.g. `lifecycle/pull-request/`, `pipeline/phases/`). Evans, _Domain-Driven Design_.                                                                                                                                                                                                                                                                                                                                                                            |
| **Wildcard-exposed boundary** | Any path reachable via a subpath-pattern `exports` entry in `package.json` — today: every path under `core/src/{types,schemas,shared,board}/` reachable via `core`'s wildcard. Node subpath patterns match multi-segment paths, so nested paths (`core/board/shortcut/labels/labels.js`) resolve too. Each reachable path is semver-locked public API.                                                                                                                                                                                         |
| **Dependency direction**      | Core ← dev ← terminal ← chief-clancy. Core imports nothing from terminal. Terminal imports from core + dev. Brief, plan, and scan are standalone (no core/terminal deps). Enforced by eslint-plugin-boundaries. Also called the **package-dependency boundary** when disambiguation is needed.                                                                                                                                                                                                                                                 |
| **Package-export boundary**   | For packages that publish a consumer API, the set of import paths cross-package consumers can resolve — defined by `package.json` `exports` (literal subpaths plus wildcards). Distinct from the **package-dependency boundary** (see "Dependency direction") which governs _which packages_ may import from which. An import path inside `exports` resolves; one outside doesn't, even if the file is physically present. Symbol visibility within a resolved entrypoint is governed by that module's own `export` statements.                |
| **Public API**                | The set of import paths exposed via `package.json` `exports` (including wildcards), plus the symbols each resolved entrypoint module exports. The authoritative package-level boundary — for published consumers, imports outside this surface break at publish time. Consistent with [CONVENTIONS.md Rule 11](CONVENTIONS.md#code-style). References to module-local exports or barrels in [CONVENTIONS.md §Export Hygiene](CONVENTIONS.md#export-hygiene) describe local export structure, not a different definition of package public API. |
| **Changeset**                 | A file describing a version bump and changelog entry, managed by `@changesets/cli`. Created with `pnpm changeset`.                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Phase validation**          | Protocol run before starting each delivery phase. Two agents (breakdown validator + DA) review the PR list for scope, ordering, and risk.                                                                                                                                                                                                                                                                                                                                                                                                      |
| **Invoke strategy**           | The pattern where core's pipeline defines pure phases, but the actual Claude CLI invocation is terminal's responsibility. Core returns what to invoke; terminal executes it.                                                                                                                                                                                                                                                                                                                                                                   |
| **DA (devil's advocate)**     | A review agent spun up as a subagent to challenge code, docs, or plans. Runs in a fresh context (not biased by having written the code). Used at phase validation, PR completion, and pre-merge. See [DA-REVIEW.md](DA-REVIEW.md) for the checklist, Required disciplines, and Severity Labels.                                                                                                                                                                                                                                                |
| **Tracer bullet TDD**         | The project's TDD approach: write one test → implement to pass → next test → repeat → refactor. Vertical slices, not horizontal. Never write all tests first then all implementation — tests written in bulk test imagined behaviour. Adapted from Matt Pocock's [`tdd`](https://github.com/mattpocock/skills/tree/main/tdd) skill. See [TESTING.md](TESTING.md).                                                                                                                                                                              |

## Roles

| Term            | Definition                                                                                                                                                                                                                        |
| --------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Implementer** | Core role. Picks up tickets from the board, implements them, delivers via PR. Commands: `/clancy:implement`, `/clancy:autopilot`, `/clancy:dry-run`.                                                                              |
| **Reviewer**    | Core role. Reviews completed work, checks quality, manages logs. Commands: `/clancy:review`, `/clancy:status`, `/clancy:logs`.                                                                                                    |
| **Setup**       | Core role. Configuration and maintenance. Commands: `/clancy:init`, `/clancy:settings`, `/clancy:doctor`, `/clancy:update-terminal`, `/clancy:map-codebase`, `/clancy:update-docs`, `/clancy:uninstall-terminal`, `/clancy:help`. |
| **Planner**     | Optional role. Generates implementation plans for tickets before coding begins. Commands: `/clancy:plan`, `/clancy:approve-plan`.                                                                                                 |
| **Strategist**  | Optional role. Decomposes vague ideas into actionable tickets via research and stakeholder grilling. Commands: `/clancy:brief`, `/clancy:approve-brief`.                                                                          |

## Delivery

| Term                   | Definition                                                                                                                                                                                                                                                                                                                                                                                                                                                          |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Board**              | The project management tool (Jira, GitHub Issues, Linear, Shortcut, Notion, Azure DevOps) where tickets live. Clancy reads from and writes to the board via API. **Optional** — Clancy also runs in local mode against plan files when no board is configured.                                                                                                                                                                                                      |
| **Local mode**         | No-board operation. `.clancy/.env` has none of the board markers, so Clancy works off local plan files in `.clancy/plans/` instead of a ticket queue. `/clancy:autopilot` and `/clancy:review` require a board; everything else works.                                                                                                                                                                                                                              |
| **Ticket**             | A unit of work. In board mode it's a row on the Kanban board. In local mode Clancy synthesises a ticket from a plan file (`--from`) to pass through the same pipeline.                                                                                                                                                                                                                                                                                              |
| **Plan file**          | A local `.md` file in `.clancy/plans/` written by `/clancy:plan --from`. Contains the same structured plan that would otherwise be posted as a ticket comment.                                                                                                                                                                                                                                                                                                      |
| **`.approved` marker** | Sibling file at `.clancy/plans/{stem}.approved` written by `/clancy:approve-plan`. Plain-text body: `sha256={hex}\napproved_at={iso-timestamp}`. **Write-side contract today** — a future runtime gate (currently deferred per `approve-plan.md`) will re-hash the plan file and refuse to run on mismatch. The verifier exists in `@chief-clancy/dev` but is not yet wired into the pipeline; until then, approval is enforced manually or by workflow convention. |
| **Parented ticket**    | A ticket that has a parent (epic in Jira, milestone in GitHub, parent issue in Linear). Delivered via PR to the epic branch.                                                                                                                                                                                                                                                                                                                                        |
| **Standalone ticket**  | A ticket with no parent. Delivered via PR directly to the base branch.                                                                                                                                                                                                                                                                                                                                                                                              |
| **Epic branch**        | A long-lived branch where child ticket PRs are merged. Named `epic/{key}` for Jira/Linear or `milestone/{slug}` for GitHub. When all children are done, the epic branch gets a PR to the base branch.                                                                                                                                                                                                                                                               |
| **Base branch**        | The branch configured as `CLANCY_BASE_BRANCH` (default: `main`). The target for standalone ticket PRs and epic PRs.                                                                                                                                                                                                                                                                                                                                                 |
| **Feature branch**     | A short-lived branch (e.g. `feature/proj-101`) created for implementing a single ticket. PRs target either the epic branch or base branch.                                                                                                                                                                                                                                                                                                                          |
| **Single-child skip**  | Optimisation: if an epic has only one child ticket, skip the epic branch overhead — deliver the child PR directly to the base branch.                                                                                                                                                                                                                                                                                                                               |
| **Epic completion**    | When all children of an epic are done (PRs merged), Clancy auto-creates a PR from the epic branch to the base branch.                                                                                                                                                                                                                                                                                                                                               |
| **Migration guard**    | Safety check: if an epic branch exists locally but not on the remote, block and show instructions to push manually.                                                                                                                                                                                                                                                                                                                                                 |

## Once Orchestrator

| Term                  | Definition                                                                                                                                                                                                                                                                                                                    |
| --------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Once**              | A single ticket execution cycle: preflight → fetch ticket → implement → deliver → log. Entry point: `/clancy:implement`.                                                                                                                                                                                                      |
| **Run**               | A batch of consecutive `Once` cycles. Board mode: `/clancy:autopilot` loops the queue until empty or `MAX_ITERATIONS` is reached. Local mode: `/clancy:implement --from .clancy/plans/ --afk` iterates approved plans in the directory sequentially.                                                                          |
| **Preflight**         | Startup checks: `.clancy/.env` exists, credentials valid, board reachable (board mode). In local mode (`--from`), preflight currently seeds a no-op board + synthetic config and does not enforce the `.approved` marker gate at runtime — that verifier is wired into the pipeline in a future PR. Runs before every ticket. |
| **Blocker check**     | Before implementing, check blocking dependencies on the board. If any blocker is incomplete, the ticket is skipped. Board mode only — local mode has no blocker graph.                                                                                                                                                        |
| **Board type**        | Unified type abstracting all board operations. Created by `createBoard()` factory — the single switch on `config.provider` in the system.                                                                                                                                                                                     |
| **Phase pipeline**    | The once orchestrator is a pipeline of 13 composable phase functions sharing a `RunContext`. Each phase returns continue or exit.                                                                                                                                                                                             |
| **Feasibility check** | After fetching a ticket, assess whether the work is achievable in the current codebase context.                                                                                                                                                                                                                               |
| **Rework**            | Automatic re-implementation triggered by PR review comments. Inline code comments always trigger; conversation comments need `Rework:` prefix.                                                                                                                                                                                |
| **Progress entry**    | A line in `.clancy/progress.txt` recording a completed action.                                                                                                                                                                                                                                                                |
| **TDD mode**          | Test-driven development mode enabled by `CLANCY_TDD=true`.                                                                                                                                                                                                                                                                    |

## Strategist

| Term               | Definition                                                                                                                                                                                                                                                                                                                                |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Brief**          | A strategic decomposition document generated by `/clancy:brief`. Contains problem statement, goals, discovery Q&A, user stories, ticket decomposition, and risks.                                                                                                                                                                         |
| **Grill phase**    | The relentless clarification step before brief generation. Resolves dependencies between decisions one by one. Zero ambiguity before a single ticket is written. _Strategist-specific — not the same as "Spec grilling" below._                                                                                                           |
| **Spec grilling**  | Adversarial review of a draft spec (PR-spec, ruleset, plan, rationale doc) for gaps, contradictions, and missing scope. Applied before the spec becomes actionable. Spawned as a subagent (writer ≠ griller); iterate until findings are nit-picks. Distinct from "Grill phase" above: same root word, scoped by the noun it attaches to. |
| **Human grill**    | Interactive grill mode (default). The strategist interviews the human relentlessly.                                                                                                                                                                                                                                                       |
| **AI-grill**       | Autonomous grill mode (triggered by `--afk` flag). A devil's advocate agent interrogates its sources.                                                                                                                                                                                                                                     |
| **Vertical slice** | A ticket that cuts through all integration layers to deliver one thin, working piece of functionality end-to-end.                                                                                                                                                                                                                         |
| **HITL**           | Human-in-the-loop. A ticket tagged as requiring human judgement during implementation.                                                                                                                                                                                                                                                    |
| **AFK**            | A ticket tagged as implementable autonomously without human intervention.                                                                                                                                                                                                                                                                 |

## Pipeline Labels

| Term                   | Definition                                                                          |
| ---------------------- | ----------------------------------------------------------------------------------- |
| **Pipeline label**     | A board label marking which stage a ticket is at in Clancy's pipeline.              |
| **`clancy:brief`**     | Label for tickets that have been briefed but not yet approved.                      |
| **`clancy:plan`**      | Label for tickets that need planning.                                               |
| **`clancy:build`**     | Label for tickets ready for implementation.                                         |
| **Label crash safety** | Labels are transitioned add-before-remove: new label added first, then old removed. |

## Review process

| Term                              | Definition                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                   |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Copilot-surrogate**             | A Claude subagent dispatched per [DEVELOPMENT.md §Post-PR flow](DEVELOPMENT.md#post-pr-flow) step 1 to run factual-claim review. Two triggers: mandatory on drift-fix PRs (any commit in the PR uses type `fix(docs)` or `fix(decisions)`), and as fallback when **Copilot-unreachable detection** fires. When both apply, it runs once and the audit comment cites the mandatory trigger. Reads each file in the PR diff at HEAD — not diff-scoped — which closes the kept-prose factual-drift class the diff-scoped DA stack systematically misses. Runs [DA-REVIEW.md §Claim-extraction pass](DA-REVIEW.md#claim-extraction-pass) + [§Multi-section internal-consistency pass](DA-REVIEW.md#multi-section-internal-consistency-pass) + [§Schema-pair check](DA-REVIEW.md#schema-pair-check). Returns findings in-band; Claude posts the audit comment. Informational — surrogate findings do not gate auto-merge. Agent: [`.claude/agents/copilot-surrogate.md`](../.claude/agents/copilot-surrogate.md). |
| **Copilot-unreachable detection** | Protocol that classifies whether a Copilot review request has silently dropped on monthly-quota / rate-limit / transient failure (Copilot posts no status on these). Two observable signals, both required (observed behaviour, not documented API contract): (a) the `gh api -X POST .../requested_reviewers` response body's `requested_reviewers` returns `[]`; (b) no `copilot_work_started` event appears in the PR timeline within ~4 min of the request. When both fire, classify UNREACHABLE and skip retry. Informational — does not bypass the [§Auto-merge criteria](DEVELOPMENT.md#auto-merge-criteria) Copilot gate, which still routes through the 10min hang fallback for no-prior-round PRs. Defined at [DEVELOPMENT.md §Post-PR flow](DEVELOPMENT.md#post-pr-flow) step 1. Triggers **Copilot-surrogate** dispatch via the fallback branch.                                                                                                                                                 |

## Reliability

| Term                   | Definition                                                                                                |
| ---------------------- | --------------------------------------------------------------------------------------------------------- |
| **Verification gate**  | Agent-based Stop hook that runs lint, test, and typecheck after implementation but before delivery.       |
| **Self-healing retry** | When verification fails, Clancy feeds errors back to Claude and retries up to `CLANCY_FIX_RETRIES` times. |
| **Lock file**          | `.clancy/lock.json` — prevents double-runs and enables crash recovery.                                    |
| **Resume detection**   | On startup, if a lock file exists and the PID is dead, Clancy resumes the crashed session.                |
| **Branch guard**       | PreToolUse hook that blocks force push, direct push to protected branches, and destructive resets.        |
| **Time guard**         | Warns at 80% and 100% of `CLANCY_TIME_LIMIT`.                                                             |
| **Cost log**           | `.clancy/costs.log` — append-only file recording duration-based token estimates per ticket.               |
| **Session report**     | `.clancy/session-report.md` — generated by the AFK loop after `/clancy:autopilot` completes.              |

## Infrastructure

| Term               | Definition                                                                                                                  |
| ------------------ | --------------------------------------------------------------------------------------------------------------------------- |
| **Hook**           | A Node.js CommonJS script that runs on Claude Code events (SessionStart, PreToolUse, etc.). Best-effort — must never crash. |
| **Board module**   | TypeScript code handling API communication with a specific board platform.                                                  |
| **Remote**         | The git hosting platform (GitHub, GitLab, Bitbucket). Detected from the git remote URL.                                     |
| **Bundle**         | The esbuild-compiled runtime scripts copied to `.clancy/` during install. Self-contained, zero npm dependency.              |
| **Command**        | A user-facing markdown file defining a slash command.                                                                       |
| **Workflow**       | An implementation-detail markdown file referenced by commands. Not directly invocable.                                      |
| **Quiet hours**    | AFK runner pauses during `CLANCY_QUIET_START`–`CLANCY_QUIET_END`.                                                           |
| **Drift detector** | PostToolUse hook that compares `.clancy/version.json` against the installed package version.                                |
